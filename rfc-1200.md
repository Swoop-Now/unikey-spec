## Abstract

Delegation is a core architectural pillar of the UniKey trust model, allowing an Issuer to grant limited, cryptographically verifiable authority to another entity (a "Delegate") without transferring identity or root control. This document specifies the invariants and packet structures required to ensure that authority can be safely distributed across a chain of actors while strictly enforcing scope reduction and temporal bounds. This mechanism is critical for the "Action Economy," where AI agents must execute tasks on behalf of human users within predefined constraints.

---

## 1. Introduction

In traditional systems, delegation often involves sharing credentials or broad "API keys," which creates massive security risks. UniKey replaces this with Chained Trust Packets. An Issuer issues a specific Delegation Packet to a Delegate; the Delegate then signs a transaction packet that references its parent. This creates a cryptographically unbreakable audit trail from the root device to the final action.

## 2. Delegation Invariants (The "Rules of the Road")

All delegation within the UniKey ecosystem MUST satisfy these four invariants:

1. Explicit Grant: Authority must be explicitly signed by the parent in a dedicated packet.
2. Scope Reduction Only: A Delegate can only possess a subset (or equal) of the parent's authority. Authority MUST NEVER expand across a delegation boundary.
3. Independent Verifiability: Any verifier must be able to validate the entire chain back to the Issuer without contacting a central server.
4. Temporal Limitation: Delegated authority MUST have an expiration timestamp equal to or sooner than its parent.

## 3. Delegation Packet Structure (Normative)

A Delegation Trust Packet is a specialized version of the standard Trust Packet defined in UniKey RFC 2001. It MUST include the following fields in the claims object:

* parent_tph: The cryptographic hash (TPH) of the immediate parent Trust Packet.
* delegated_to: The identity (subject) of the authorized Delegate.
* scope: An array of specific actions the Delegate is permitted to take.
* constraints: (Optional) Numerical or logic-based limits (e.g., amount_max: 50).

## 4. Chain Validation Rules

Verifiers MUST apply the following recursive logic to any delegated action:

1. Independent Validation: Each packet in the chain must pass its own signature and temporal checks.
2. Hash Linking: The parent_tph of packet $N$ must exactly match the hash of packet $N-1$.
3. Monotonicity: The scope of packet $N$ must be a mathematical subset of packet $N-1$.
4. Expiration: If any packet in the chain is expired, the entire chain MUST be rejected.

## 5. Revocation Propagation

UniKey enforces "Upstream Revocation." If an Issuer revokes their public key or nullifies a specific parent packet (as per UniKey RFC 3002), all downstream delegated packets automatically lose their validity. Verifiers MUST check the revocation status of the root Issuer at every verification event.

## 6. Example: The "Payment Agent" Flow

1. Issuer (User Device): Issues a packet to PSP (Delegate) with scope ["pay:max_100"].
2. PSP (Delegate): Issues a packet to AI Agent (Sub-Delegate) with scope ["pay:max_50"].
3. AI Agent: Signs a transaction for $49.99.
4. Verifier (Merchant): Confirms the $49.99 is $\leq$ 50, and 50 is $\leq$ 100. Verification succeeds.

## 7. Security Considerations

* Scope Escalation: Prevented by the monotonicity requirement in Section 4.
* Stale Delegation: Mitigated by the mandatory expires_at field and Upstream Revocation.
* Replay: Each packet in the chain carries its own unique nonce.

## 8. Conformance

An implementation is conformant with UniKey RFC 1200 if it:

1. Correctly parses and validates parent_tph links.
2. Strictly enforces that a Delegate's scope is a subset of the Issuer's scope.
3. Fails validation if any link in the delegation chain is missing, expired, or improperly signed.
