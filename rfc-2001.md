## Abstract

The UniKey Trust Packet is a universal, transport-agnostic cryptographic container for proving identity, authority, and intent. It enables any verifier—without prior integration, shared state, or platform dependency—to deterministically decide whether an action is authorized. Trust Packets are designed to survive hostile or lossy internet transports, remain independently verifiable by any party, and fail closed under ambiguity or attack. This specification defines the invariant foundation for secure agent action, payments, and authentication across the open internet.

---

## 1. Introduction

A Trust Packet is a self-contained, cryptographically signed data structure that asserts identity, authority, and intent. Unlike session-based authentication, a Trust Packet carries its own proof of validity, allowing it to be transmitted across heterogeneous environments (email headers, APIs, QR codes, or message queues) while maintaining cryptographic integrity.

## 2. Scope

This document specifies the canonical packet structure, required fields, and the deterministic rules for canonicalization, hashing, and signature encoding. It further defines multi-signature support and delegation chain semantics. It does not define specific transport protocols or business-layer policy interpretation.

## 3. Terminology

The key words MUST, MUST NOT, SHOULD, SHOULD NOT, and MAY in this document are to be interpreted as described in RFC 2119.

- TPH (Trust Packet Hash): The unique cryptographic digest of a canonicalized packet.
- Verifier: Any entity that receives a Trust Packet and evaluates its signatures and claims.

## 4. Requirements (Normative)

1. Implementations MUST ensure packets remain verifiable across lossy transports.
2. Verification MUST fail closed; any partial validation or ambiguity MUST result in a rejected packet.
3. Canonicalization MUST be applied prior to hashing to ensure deterministic results across different platforms.

## 5. Specification

### 5.1 Logical Structure

A Trust Packet is composed of four top-level components:

- Header: Protocol metadata.
- Claims: Identity, authority, and context assertions.
- Payload: Action-specific data (opaque to the protocol).
- Signatures: An array of one or more cryptographic attestations.

### 5.2 Required Fields

| Field | Description |
|-------|-------------|
| tp_version | Format version (e.g., "1.0") |
| packet_id | Globally unique identifier (UUID) |
| issued_at | UTC timestamp (ISO 8601) |
| expires_at | UTC expiration timestamp |
| subject | The identity being asserted |
| issuer | The signing authority |
| scope | Array of allowed actions |
| signatures | Array of cryptographic signature objects |

### 5.3 Canonicalization Rules (UniKey-JSON-v1)

To ensure identical hashes across different systems, the following rules MUST be applied:

1. Lexicographical Ordering: All fields MUST be ordered alphabetically by key name.
2. Encoding: MUST use UTF-8 without a Byte Order Mark (BOM).
3. Whitespace: MUST remove all insignificant whitespace and line breaks.
4. Binary Data: MUST be encoded using base64url without padding.

### 5.4 Hashing and Signing

1. Construct the packet excluding the signatures field.
2. Apply UniKey-JSON-v1 canonicalization.
3. Compute the digest using the specified hash function (default: SHA-256) to produce the TPH.
4. All signatures in the signatures array MUST be computed over this TPH.

## 6. Verification Procedure (Normative)

```
function verifyTrustPacket(packet):
  if packet.tp_version not supported:
    return TP-005 # Unsupported Version

  canonical = canonicalize(packet minus signatures)
  tph = hash(canonical)

  for sig in packet.signatures:
    pubkey = resolvePublicKey(sig.signer, sig.key_id)
    if not verifySignature(pubkey, tph, sig.signature):
      return TP-002 # Signature Mismatch

  if now < packet.issued_at or now > packet.expires_at:
    return TP-003 # Temporal Failure

  if replayDetected(packet.nonce):
    return TP-001 # Replay/Canonicalization Error

  return VALID
```

## 7. Operational Constraints

- Typical Size: < 4 KB
- Maximum Size: 16 KB (Packets exceeding this MAY be rejected).
- Latency Target: Verification SHOULD complete in < 5ms on commodity hardware.

## 8. Error Handling

| Code | Condition | Action |
|------|-----------|--------|
| TP-001 | Canonicalization/Replay Failure | Reject |
| TP-002 | Signature Validation Failure | Reject |
| TP-003 | Temporal Validity (Expired/Future) | Reject |
| TP-004 | Delegation Chain Failure | Reject |
| TP-005 | Unsupported Version/Algorithm | Reject |

## 9. Security Considerations

Trust Packets are designed for adversarial environments. Verifiers MUST assume transports are lossy and intermediaries are malicious.

- Integrity: Guaranteed by TPH signatures.
- Replay Resistance: Enforced via nonce and strict expires_at checks.
- Scope Containment: Delegation chains MUST only narrow authority; they can never expand it.

### 9A. Reference Threat Scenarios

- TS-1 (Replay): Attacker replays a valid packet. Mitigation: Nonce uniqueness and temporal checks.
- TS-2 (Escalation): A delegated packet attempts to add scopes. Mitigation: Verifiers enforce monotonic scope reduction.
- TS-3 (Mutation): Whitespace altered in transit. Mitigation: Canonicalization ensures the signature remains valid.

## 10. Transport Considerations

Trust Packets are transport-agnostic. They MAY be conveyed via SMTP, HTTPS, Message Queues, or QR/NFC handoff. No transport-specific metadata is considered authoritative; security is derived solely from the packet itself.

## 11. Licensing and Contractual Reference

This specification is intended for normative reference in commercial agreements.
"Trust Packet" means a cryptographically signed data structure conforming to UniKey RFC 2001. Any implementation claiming UniKey compatibility MUST conform to this specification.

## 12. Registry and Extensibility

- Canonicalization Registry: unikey-json-v1 (Default).
- Algorithm Agility: Verifiers MUST support algorithm identifiers and MUST reject deprecated algorithms (e.g., SHA-1).

## Appendix B: Conformance and Invariant Mapping

An implementation is conformant if and only if it:

1. Produces packets matching the lexicographical order defined in Section 5.
2. Rejects any packet failing temporal or signature validation.
3. Ensures that trust decisions occur below the application layer and above the transport layer.
