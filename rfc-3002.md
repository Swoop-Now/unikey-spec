## Abstract
This document specifies a global, deterministic profile for cryptographic key discovery, validation, and trust using the Domain Name System (DNS) and DomainKeys Identified Mail (DKIM) as the public trust fabric. It defines how UniKey implementations discover, validate, cache, rotate, and revoke public keys used to verify Trust Packets. By treating DNS as a public key directory and DKIM as a signature container, this profile enables infrastructure-level trust without bilateral integration or shared state.

---

## 1. Introduction

UniKey leverages globally deployed DNS infrastructure to distribute public keys at internet scale. Unlike baseline DKIM, which is optimized for email delivery and often permits "fail-soft" scenarios, the UniKey profile requires strict, deterministic validation. This document defines the roles of Issuers and Delegates and the mandatory DNS record formats required for UniKey conformance.

## 2. Key Roles

### 2.1 Issuer (Root Authority)
The Issuer is the entity asserting a root identity (e.g., a merchant domain or device manufacturer).
- Responsibilities: Publishes public keys via DNS, signs Trust Packets, and maintains key rotation/revocation discipline.

### 2.2 Delegate (Limited Authority)

A Delegate is an entity (e.g., a PSP) granted specific authority by an Issuer through a delegation Trust Packet.
- Requirements: Must sign packets with its own key and reference the Issuer's delegation chain. Delegation MUST narrow authority and MUST be independently verifiable.

## 3. DNS Record Formats & Tag Profile

Public keys MUST be published using DKIM-compatible DNS TXT records at <selector>._domainkey.<domain>.

### 3.1 Normative Tag Profile

To ensure deterministic verification, UniKey verifiers MUST strictly adhere to this tag subset:
- v (Version): MUST be DKIM1.
- k (Key Type): MUST be specified (e.g., rsa, ed25519).
- p (Public Key): MUST be base64url encoded.
- Disallowed Tags: g, h, n, and all experimental tags. Presence of these tags MUST cause verification failure.

## 4. Selector Discipline

Selectors identify specific public key versions.
1. Selectors MUST be unique per active key.
2. Reuse of a selector for a different key is strictly prohibited.
3. Example: 2026q1._domainkey.example.com

## 5. Cryptographic Minimums

Verifiers MUST reject any keys failing to meet these minimum strengths:
- RSA: 2048-bit minimum.
- ECC: P-256 or Ed25519.
- Hash: SHA-256 or stronger.

## 6. Caching & TTL Guidance

Verifiers MAY cache keys but MUST honor DNS TTLs.
- Maximum Cache Lifetime: 24 hours.
- Revalidation: Must occur on TTL expiry or upon detection of a verification anomaly.
- Cache entries MUST be bound to the specific (selector, domain) pair.

## 7. Key Rotation & Revocation

### 7.1 Rotation
Issuers SHOULD rotate keys quarterly. New selectors MUST be published and propagated through the DNS overlap window (at least equal to the TTL) before being used to sign production packets.

### 7.2 Revocation
Revocation is executed by:
1. Removing the DNS TXT record, OR
2. Setting the p= tag to an empty value (p=).

Verifiers MUST treat missing or empty keys as invalid.

## 8. DNS Hardening Requirements

Implementations MUST apply the validation measures defined in UniKey RFC 3001, including multi-resolver correlation and change detection. DNSSEC MAY be used as an auxiliary signal but MUST NOT replace multi-path validation.

## 9. Failure & Fallback Behavior

UniKey is a fail-closed system. If a key is not found, is malformed, or exhibits DNS anomalies, the authentication decision MUST be rejected. There is no "silent downgrade" to weaker security mechanisms.

## 10. Contrast with Baseline DKIM

| Feature | Baseline DKIM | UniKey Profile |
|---------|---------------|----------------|
| Primary Use | Email Authentication | Global Identity Anchor |
| Validation | Fail-Soft (Spam folder) | Fail-Closed (Reject Action) |
| Key Discovery | Header-driven | Infrastructure-driven |
| Security | Heuristic | Deterministic |

## 11. Security Considerations

This profile addresses DNS poisoning, key substitution, and delegation abuse. By treating DNS as a distributed, append-only trust directory and enforcing strict selector discipline, UniKey ensures that identity remains anchored to the domain owner's intent.

## 12. Conformance

An implementation is conformant with UniKey RFC 3002 if it:
1. Discovers keys using the <selector>._domainkey path.
2. Enforces the Normative Tag Profile (Section 3.1).
3. Rejects keys below the minimum cryptographic strength (Section 5).
4. Applies fail-closed logic to all DNS anomalies.

## Appendix A: Cross-Domain Delegation Flow (Normative)

1. Issuer (merchant.com) publishes DKIM key via DNS.
2. Issuer issues a Delegation Trust Packet to Delegate (psp.com).
3. Delegate signs a transaction packet referencing the Delegation Packet.
4. Verifier validates Delegate's key via DNS, then traces the chain back to the Issuer.
